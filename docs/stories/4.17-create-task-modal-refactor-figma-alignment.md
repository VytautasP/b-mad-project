# Story 4.17: Create Task Modal - Refactor & Figma Alignment

## Status

Approved

## Story

**As a** TaskFlow user,
**I want** the Create Task modal to match the approved Figma design,
**so that** creating a task feels polished, predictable, and consistent across the shellbar, task list, and project entry points.

## Design Reference

- **Figma file:** TaskFlow - Create Task Modal via figma_local MCP server
- **Node:** `170:144` (`Modal Content`)
- **Supporting nodes:** `170:145` (Header), `170:154` (Body/Form), `170:226` (Footer)
- **Primary UX reference:** `docs/front-end-spec.md`
- **Triggered from:** task list create CTA, shellbar Create Task button, project-level create action

## Figma Design Summary

The approved design defines a centered dialog with a clear three-part structure: **header**, **form body**, and **footer actions**. The current Angular `TaskFormComponent` already supports the necessary fields and submission flow, but the modal presentation is still close to default Angular Material styling. This story refactors the dialog to match the approved Figma layout while preserving the existing create-task behavior.

### Modal Container (`170:144`)
- Width: `672px`
- Height: auto / hug content (`631px` in the source frame)
- Background: `#FFFFFF`
- Border: `1px solid #E2E8F0`
- Border radius: `8px`
- Shadow: `0 25px 50px -12px rgba(0, 0, 0, 0.25)`
- Layout: vertical, clipped content, no internal outer padding on the shell itself

### Header (`170:145`)
- Height: `77px`
- Padding: `20px` vertical, `32px` horizontal
- Bottom border: `1px solid #E2E8F0`
- Title: `Create New Task`
- Title typography: Manrope 700 / `18px` / `28px` / `-0.45px` letter-spacing / `#111418`
- Right-aligned close affordance above the form body

### Body/Form (`170:154`)
- Padding: `24px` top, `32px` left/right, `40px` bottom
- Vertical gap between groups: `24px`
- Field order:
  1. Task Name (full width)
  2. Description (full width textarea)
  3. Three-column planning row: Due Date, Priority, Status
  4. Type (full width)
- Labels: Manrope 600 / `14px`
- Desktop planning row: three equal-width columns with `24px` gap
- Tablet/mobile: planning row stacks vertically in the same field order

### Footer (`170:226`)
- Height: `81px`
- Padding: `20px` vertical, `32px` horizontal
- Background: `#F6F7F8`
- Top border: `1px solid #E2E8F0`
- Right-aligned action group with `12px` gap
- Secondary button: `Cancel`
  - Text color: `#64748B`
  - Border radius: `8px`
  - Padding: `20px` horizontal / `10px` vertical
- Primary button: `Create Task`
  - Background: `#137FEC`
  - Text color: `#FFFFFF`
  - Border radius: `8px`
  - Padding: `32px` horizontal / `10px` vertical
  - Shadow: layered blue button shadow per the Figma source

## Acceptance Criteria

1. **All create entry points** open the same refactored Create Task dialog from `TaskFormComponent`: task list, shellbar, and project view.
2. **Dialog container** matches the Figma shell: centered, `672px` desktop width, white background, `8px` radius, `1px` `#E2E8F0` border, and `0 25px 50px -12px rgba(0, 0, 0, 0.25)` shadow.
3. **Dialog header** is implemented as a distinct section with `32px` horizontal / `20px` vertical padding, bottom divider, title "Create New Task", and a close button aligned right.
4. **Dialog body** follows the Figma field order exactly: Task Name, Description, Due Date / Priority / Status row, then Type.
5. **Task Name** remains required and is visually emphasized as the first field in the modal.
6. **Description** renders as a full-width textarea directly beneath Task Name with the updated spacing and label treatment.
7. **Due Date, Priority, and Status** render in a three-column row at desktop widths and stack vertically on narrower screens without changing field order.
8. **Type** renders as a full-width field beneath the planning row.
9. **Footer section** is visually separate from the form body with `#F6F7F8` background, top divider, right-aligned actions, and visible `Cancel` plus `Create Task` buttons.
10. **Primary action styling** matches the approved Figma treatment: brand-blue fill, white text, `8px` radius, stable width during loading, and stronger visual emphasis than Cancel.
11. **Cancel and close button behavior** both dismiss the dialog and return focus to the invoking control when the form is not actively submitting.
12. **Submit success behavior** remains unchanged: the dialog closes, the originating surface refreshes, and success feedback appears.
13. **Submit failure behavior** remains in-modal: entered values are preserved, an inline error message appears within the form, and the user can retry without re-entering the task.
14. **Accessibility requirements** are met for the modal: dialog name, visible labels, keyboard traversal in visual order, Escape behavior, focus return, icon button accessible names, and visible focus states.
15. **Typography and tokens** match the approved spec for this modal: Manrope inside the dialog, `#111418` primary text, `#64748B` secondary text, `#137FEC` primary CTA, `#E2E8F0` structural borders.
16. **Responsive behavior** keeps the modal usable from `320px` upward, using viewport-fit width with safe margins and stacked planning controls on smaller breakpoints.
17. **Shared-component regressions are avoided:** edit mode and embedded usage of `TaskFormComponent` continue to function, with correct labels and submission behavior.
18. **Frontend tests** are updated to cover the refactored create dialog layout, close/cancel behavior, loading/error states, and responsive field grouping.

### Current State

- `TaskFormComponent` currently handles both create and edit mode in a single standalone Angular component.
- The current create dialog uses default Angular Material form-field presentation with a `600px` dialog width and no dedicated modal shell matching the approved Figma structure.
- The current dialog layout stacks all fields vertically and only shows `Cancel` in embedded mode, not in the main create dialog flow.
- Escape close behavior is implemented directly on the component and should be preserved, but brought into alignment with the refactored dialog states.
- Create dialog openers currently exist in:
  - `src/app/features/tasks/task-list/task-list.component.ts`
  - `src/app/shared/components/shellbar/shellbar.ts`
  - `src/app/features/project/project.ts`

### Angular Implementation Constraints

- Keep the refactor frontend-only. No API contract or DTO changes are required.
- Maintain Angular standalone component structure and the current reactive form approach.
- Continue routing all persistence through `TaskService`; do not introduce direct HTTP calls from the component.
- Keep the component focused and avoid expanding this story into edit-flow redesign or new task fields.
