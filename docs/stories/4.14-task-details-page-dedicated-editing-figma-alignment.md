# Story 4.14: Task Details Page — Dedicated Editing Workspace & Figma Alignment

## Status

Approved

## Story

**As a** TaskFlow user,
**I want** a dedicated task details page that matches the approved full-details design,
**so that** I can review, edit, and manage a task in one focused workspace without modal friction.

## Implementation Direction

- The existing `task-detail-dialog` / Quick Inspect experience must remain intact as the lightweight modal introduced in prior stories.
- This story is specifically for creating a **new dedicated full-task editing component/workspace** for the route-based experience, rather than expanding the current Quick Inspect component into the full editor.

## Design Reference

- **Figma file:** TaskFlow — Task Full Details via figma_local MCP server
- **Node:** `1:495` (`Task - Full details` page)
- **Key layout frames:** `7:281` Header, `7:317` Sidebar, `7:351` Main, `7:387` Two-column content container
- **Existing route:** `/tasks/:id`

## Figma Design Summary

The Figma page defines the task details experience as a full authenticated workspace rather than a long utility page. The layout uses the existing app shell pattern: **64px top header**, **256px left sidebar**, and a **light-gray main canvas** with a centered content container.

### Overall Page Layout

- **Shellbar/Header (`7:281`)**
  - Height: `64px`
  - Background: white
  - Bottom border: `1px solid #E2E8F0`
- **Sidebar (`7:317`)**
  - Width: `256px`
  - White background with right border
- **Main area (`7:351`)**
  - Background: `#F6F7F8`
  - Padding: `32px`
- **Content container (`7:387`)**
  - Max content width: `960px`
  - Desktop layout: **632px primary column** + **304px secondary column**
  - Gap between columns: `24px`

### Primary Column

#### 1. Overview Card (`7:389`)
- White surface, `12px` radius, `1px solid #E2E8F0`, subtle shadow `0 1px 2px rgba(0,0,0,0.05)`
- Padding: `32px`
- Large title block using Manrope bold display typography
- Supporting multi-line description directly below the title
- Compact top-right utility action button

#### 2. Details Card (`7:398`)
- White card, `12px` radius, border + subtle shadow
- Padding: `24px`
- Section header: **Details**
- Right-aligned compact pill/button for edit-oriented action
- Content is a stacked list of editable task facts instead of the current generic property grid
- Intended to make status, priority, due date, type, and assignment information easier to scan and update

#### 3. Attachments Card (`7:443`)
- White card, `12px` radius, border + subtle shadow
- Padding: `24px`
- Header text: **Attachments (3)**
- Content is a row of compact attachment tiles/cards
- The design clearly reserves space for file context inside the task details workflow

### Secondary Column

#### 4. Time Tracking Card (`7:475`)
- White card with clipped rounded corners
- Top section uses a soft blue-to-white gradient
- Header label: **Time Tracking**
- Right-side timer pill displaying formatted elapsed time (example in design: `04:12:35`)
- Action row contains:
  - **Start** button: green filled, `8px` radius
  - **Stop** button: white outline, `8px` radius
- Lower section contains a stacked summary/list area for time details

#### 5. Secondary Details Card (`7:521`)
- White card, `12px` radius, border + subtle shadow
- Header text: **Details** with compact overflow/menu affordance
- Content is a vertical stack of compact metadata rows separated with consistent spacing and a divider

#### 6. Activity Card (`7:582`)
- White card, `12px` radius, border + subtle shadow
- Header text: **Activity** with right-aligned **View all** link
- Scrollable feed area in the middle
- Bottom composer area with textarea + action row, indicating inline activity/comment entry from the task page

## Acceptance Criteria

1. The existing `/tasks/:id` route is retained, but the page is refactored to match the Figma full-details workspace layout with a centered 960px content container and a 632px / 304px two-column desktop split.
2. The top content section renders as a prominent overview card with the task title, description, and a compact utility action, matching the spacing, radius, border, and typography defined in Figma.
3. The current property-grid presentation is replaced by a dedicated **Details** card that presents editable task metadata in a cleaner stacked layout optimized for task editing and review.
4. A **new dedicated full-task editing component** is implemented for the route-based full-details workflow; the existing Quick Inspect modal/component is not repurposed into the full editor.
5. The **Edit Details** action from the task details page remains available and is surfaced prominently within the Figma-aligned details section, reusing `TaskFormComponent` only if it fits cleanly inside the new dedicated editing workspace.
6. The existing Quick Inspect modal remains summary-focused and behaviorally unchanged except for any navigation hooks needed to open the new full editing page.
7. The right rail includes a Figma-aligned **Time Tracking** card with current elapsed/summary time display and clear start/stop actions that preserve existing timer behavior.
8. The right rail includes a compact secondary metadata/details card for supplemental task facts and quick actions.
9. The page includes an **Activity** section aligned to the Figma right-rail card pattern, with recent activity/comment context and a clear path to view the full thread/feed.
10. An **Attachments** section is visually implemented per Figma; because no attachment model or API is defined in the current architecture docs, the section must degrade gracefully using an empty state or scaffolded read-only presentation without broken upload/download behavior.
11. Existing integrations continue to work: assignees, time logging, comments, activity history, timer state, and navigation from Quick Inspect to full details.
12. Loading, empty, and error states are updated so they fit inside the new card layout rather than falling back to generic page blocks.
13. Responsive behavior collapses the two-column layout into a single-column stack on narrower viewports while preserving the visual hierarchy of overview → details/attachments → right-rail cards.
14. Accessibility is preserved or improved: proper heading order, landmark usage, keyboard reachability for actions, visible focus styles, and semantic labels for icon-only controls.
15. The refactor does not regress the Quick Inspect modal’s **Open Full Details** flow introduced in Story 4.13, and Quick Inspect remains the lightweight summary view.
16. Component/unit tests cover the updated route render, new dedicated editor entry points, Quick Inspect-to-full-page navigation, timer action states, and key conditional states introduced by the new layout.

## Dev Notes

- **Architecture / implementation guidance**
  - Frontend pages are Angular standalone components and route-driven views; continue implementing the task details page as a standalone route component. [Source: `docs/architecture/frontend-architecture.md`]
  - Frontend data access must stay inside services; do not move API calls directly into templates or ad hoc helpers. [Source: `docs/architecture/12-coding-standards.md`]
  - The task data model includes `name`, `description`, `dueDate`, `priority`, `status`, `type`, `assignees`, and rollup time fields. The current repo also exposes `createdByUserName`, `directLoggedMinutes`, `childrenLoggedMinutes`, and `totalLoggedMinutes`; retain those working contracts during the UI refactor. [Source: `docs/architecture/data-models.md`, observed current model in `src/app/shared/models/task.model.ts`]
  - Time tracking, comments, and activity are already established feature areas and should be reused rather than reimplemented. [Source: `docs/architecture/components.md`, `docs/architecture/core-workflows.md`]
  - Unit tests should use the existing Angular testing stack (Jasmine + Karma). Components with business logic require tests. [Source: `docs/architecture/tech-stack.md`, `docs/architecture/12-coding-standards.md`]

- **Observed current implementation**
  - Current route: `src/app/app.routes.ts`
  - Current page files: `src/app/features/tasks/task-details-page/task-details-page.ts`, `.html`, `.css`
  - Quick Inspect modal entry point: `src/app/features/tasks/task-detail-dialog/task-detail-dialog.ts`
  - Related openers include task list, dashboard/tree, and timeline contexts.
  - `task-detail-dialog` is currently the Quick Inspect task component and should stay summary-focused; do not expand it into the full editing workspace.

- **Project structure notes**
  - Architecture docs refer to a `task-detail/` feature structure, but the current repository already uses `task-details-page/` and `task-detail-dialog/`. Keep the existing repo structure unless the developer needs a contained child-component extraction for maintainability.
  - Prefer creating a new dedicated route component or nested feature component for the full editor and keeping `task-detail-dialog` isolated as the quick-inspect surface.
  - Coding standards cap files at roughly 300 lines. If the refactor pushes the page beyond that limit, split UI sections into small presentational child components within the same feature area. [Source: `docs/architecture/12-coding-standards.md`]

- **Scope/constraint note**
  - No attachment entity, API contract, or documented workflow currently exists in the architecture set. The Figma attachments card should therefore be treated as a UI scaffold/empty state in this story, not as a full file-management feature.
