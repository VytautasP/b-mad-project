# Story 2.10: Time Logs View — Full-Stack Implementation

## Status

Ready for Review

## Story

**As a** freelancer tracking billable work,
**I want** a dedicated Time Logs page showing all time entries across tasks with filtering, summary stats, and CSV export,
**so that** I can review, audit, and report on my tracked hours efficiently.

## Backend Gap Analysis

The existing backend has a TimeEntry entity, service, and a partially-implemented controller (DELETE only). The following **new backend work** is required to support this view:

### Existing Backend Assets

- **Entity:** `TimeEntry` (Id, TaskId, UserId, Minutes, EntryDate, Note, EntryType, CreatedAt)
- **DTOs:** `TimeEntryCreateDto`, `TimeEntryResponseDto` (with UserName), `PaginatedResultDto<T>`
- **Service:** `ITimeEntryService` with `LogTimeAsync`, `GetTaskTimeEntriesAsync`, `DeleteTimeEntryAsync`
- **Repository:** `ITimeEntryRepository` with CRUD + `GetTotalMinutesByTaskIdAsync`
- **Controller:** `TimeEntriesController` — only DELETE endpoint implemented
- **Route:** `/logs` route and sidebar nav entry already exist; `LogsComponent` is a placeholder

### Required Backend Changes

1. **Add `IsBillable` field** to `TimeEntry` entity (bool, default false) + DB migration
2. **Add `IsBillable`** to `TimeEntryCreateDto` and `TimeEntryResponseDto`
3. **Add `TaskName` and `ProjectName`** to `TimeEntryResponseDto` (resolved via Task navigation + parent hierarchy)
4. **New endpoint: `GET /api/timeentries`** — paginated list of all user time entries with query params:
   - `page` (int, default 1)
   - `pageSize` (int, default 10)
   - `startDate` / `endDate` (DateTime, optional — date range filter)
   - `userId` (Guid, optional — admin/team filter; default = current user)
   - `projectId` (Guid, optional — filter by root-level Project task)
   - `isBillable` (bool?, optional)
   - `search` (string, optional — searches Note text)
   - Returns `PaginatedResultDto<TimeEntryResponseDto>`
5. **New endpoint: `GET /api/timeentries/summary`** — aggregated stats:
   - Query params: `startDate`, `endDate`, `userId` (same filters as list)
   - Returns: `{ totalMinutes, billableMinutes, nonBillableMinutes, previousPeriodTotalMinutes, previousPeriodBillableMinutes, previousPeriodNonBillableMinutes }` for percent-change calculation
6. **New endpoint: `GET /api/timeentries/export`** — CSV download with same filter params as list
7. **Complete controller:** Wire POST (create), GET by ID, and PATCH (update) endpoints already defined in the API spec
8. **Repository:** Add `GetPaginatedAsync` with filtering, `GetSummaryAsync` for aggregation, and `GetForExportAsync`
9. **Service:** Add corresponding methods to `ITimeEntryService` and `TimeEntryService`

## Acceptance Criteria

### Backend

1. `TimeEntry` entity extended with `IsBillable` (bool, default false); EF migration created and applied.
2. `GET /api/timeentries` returns a `PaginatedResultDto<TimeEntryResponseDto>` supporting `page`, `pageSize`, `startDate`, `endDate`, `userId`, `projectId`, `isBillable`, and `search` query parameters.
3. `TimeEntryResponseDto` includes `TaskName` (string) and `ProjectName` (string, resolved from nearest ancestor with `TaskType.Project`; empty if none).
4. `TimeEntryResponseDto` includes `IsBillable` (bool).
5. `GET /api/timeentries/summary` returns total, billable, and non-billable minutes for the requested period **and** the equivalent previous period (for percent-change badges).
6. `GET /api/timeentries/export` returns a `text/csv` file with columns: Date, Task, Project, Description, Duration, Billable — filtered by the same query parameters as the list endpoint.
7. `POST /api/timeentries` (create) and `PATCH /api/timeentries/{id}` (update) endpoints are wired in the controller, delegating to existing service methods.
8. Authorization enforced: users can only view/modify their own entries (admin/team filter deferred to a future story).
### Frontend

9. `LogsComponent` replaced with full implementation matching the Figma design.
10. **Page heading** renders "Time Logs" as `h2` with right-aligned "Export CSV" (secondary) and primary action button.
11. **Stats cards row** displays three summary cards (Total Hours This Week, Billable Hours, Non-Billable Hours) with values from the summary endpoint and percent-change badges.
12. **Filter bar** renders date-range picker (defaults to "This Week"), "All Users" dropdown, "Project" dropdown, and "More" dropdown — each filter triggers API re-fetch.
13. **Data table** renders columns: Checkbox, Date, Task/Project, Description, Time, Billable, Actions — matching Figma styling.
14. **Date column** shows relative label ("Today", "Yesterday") or formatted date as primary line, with time as secondary line.
15. **Task/Project column** shows task name (primary) and project name (secondary) in a two-line cell.
16. **Time column** displays formatted duration (e.g. "4h 30m") inside a styled pill/badge.
17. **Billable column** shows a checkmark icon for billable entries, empty for non-billable.
18. **Actions column** shows edit/delete/more icon buttons that appear on row hover (`opacity: 0` → `opacity: 1`).
19. **Pagination footer** shows "Showing X of Y entries" with page navigation controls.
20. **Export CSV** button triggers file download via the export endpoint with current filter state.
21. **Responsive:** Table layout at desktop; simplified card/list at mobile breakpoint.
22. **Existing functionality preserved:** Sidebar "Logs" link continues to work; timer widget and other time tracking features unaffected.

- **Project resolution approach:** The `Task` entity has `TaskType` enum (`Project = 0, Milestone = 1, Task = 2`) and `ParentTaskId`. To find the "Project" for a time entry, walk `Task → ParentTask → ...` until finding one with `TaskType == Project`. Use a CTE or recursive include if performance is a concern.
- **Date range "previous period":** For the summary percent-change, if current period is "This Week" (Mon–Sun), previous period is the prior week. Use the same duration offset (e.g. 7 days back for a week filter).
- **IsBillable default:** New field defaults to `false`. Existing time entries will all be non-billable until manually updated — this is acceptable for first release.
- **CSV export** should stream results for large datasets. For MVP, buffered in-memory is acceptable.
- Reuse existing design tokens and component patterns from the task list view (card wrapper, table headers, pagination footer, pill badges).
