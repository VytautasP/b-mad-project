# Story 4.16: Timeline View Refactor — Figma Alignment

## Status

Approved

## Story

**As a** task manager,
**I want** the Timeline view to display a Gantt-style chart with a left-hand task list panel and a right-hand date-range chart matching the approved Figma design,
**so that** I can visualise task durations and dependencies across time in a polished, design-consistent interface that replaces the current `vis-timeline` implementation.

### Key Architectural Decisions

1. **vis-timeline: Keep or Replace — Two Approaches**

   The Figma design shows a custom Gantt layout (separate left task-list panel + right chart), which diverges from vis-timeline's built-in layout. Two viable approaches exist:

   | | **Approach A — Hybrid (Recommended)** | **Approach B — Full Custom** |
   |---|---|---|
   | **Summary** | Keep `vis-timeline` for the right chart panel only. Build a custom left task-list panel. Sync scroll between them. Apply heavy CSS theming to vis-timeline items/bars. | Remove `vis-timeline` entirely. Build both panels and the Gantt bar renderer from scratch in Angular + CSS Grid. |
   | **Pros** | Drag-to-resize bars, zoom/pan, date-axis math, snapping — all free. Proven accessibility (keyboard nav). Significantly less custom code for the chart. Lower risk. | Pixel-perfect Figma match with no library constraints. Smaller bundle. No vis-timeline CSS overrides to maintain across upgrades. |
   | **Cons** | vis-timeline's DOM structure may fight you on some Figma details (bar label alignment, exact border-radius rendering). CSS overrides are fragile across library updates. vis-timeline's built-in groups sidebar must be hidden and replaced by the custom left panel. | High engineering effort — must implement date-column calculations, bar positioning, drag-to-resize, keyboard navigation, and scroll sync from scratch. High risk of edge-case bugs (timezone math, overlapping bars, resize handles). |
   | **Effort** | Medium | High |
   | **Risk** | Low–Medium | Medium–High |

   **Recommendation:** Start with **Approach A** (hybrid). Use vis-timeline with `groupsVisibility: false` (hide its built-in groups), render it inside the right chart panel `<div>`, and build the left task-list panel as a standalone Angular component. Apply CSS `::ng-deep` overrides (or `ViewEncapsulation.None` scoped to the chart) to theme bars to match Figma. If vis-timeline proves too resistant to the required visual changes during implementation, pivot to Approach B.

2. **Split-panel layout** — Left fixed-width task list (300px) + right flexible chart area with synchronized vertical scrolling.
3. **Grouped tasks** — Tasks are grouped by project/phase (e.g. "1. Discovery Phase"). The API should return tasks with `parentTaskId` or a `group` field. If grouping isn't available, group by status or project.
4. **Horizontal scroll** — The chart area scrolls horizontally (overflow-x) while the task list stays pinned. Date header scrolls with the chart.
5. **Reuse `TaskService`** — Continue using `getTimelineTasks()` for data. Extend if needed to return grouping metadata.
6. **Shared components** — Shellbar and Sidebar are already rendered by `app.html` — no changes needed.
7. **Design tokens** — Reuse status colours, typography, and border tokens established in Stories 4.10–4.15.

## Backend Considerations

The current `getTimelineTasks()` API returns a flat list of tasks with `startDate`, `endDate`, and `status`. To support the Figma design's grouped view:

1. **Task grouping** — If the backend already returns `parentTaskId` (from the task hierarchy), the frontend can build groups. If not, a minor backend change may be needed:
   - **Option A (preferred):** Add a `groupName` or `phase` field to the timeline response DTO so the frontend can group rows without extra logic.
   - **Option B:** Return tasks with `parentTaskId` populated and let the frontend derive groups from parent tasks.
   - **Option C:** No backend change — group tasks by `status` or simply show a flat list initially, add grouping in a follow-up.

2. **Date range columns** — The frontend calculates column boundaries (e.g. weeks) client-side from the `startDate`/`endDate` query params. No backend change needed.

3. **Bar positioning** — Each task's horizontal position and width are calculated from its `dueDate` (or `startDate` + duration) relative to the visible date range. The existing `Task` model has `dueDate` and `createdAt` — if `startDate` is missing, use `createdAt` as fallback.

**Recommendation:** Start with **Option C** (no backend change) for this story. Group by `parentTaskId` if available, or render a flat list. Defer phase/project-level grouping to a follow-up story.

## Acceptance Criteria

1. **Toolbar** renders with view-mode toggle (Day/Week/Month), date navigation (prev/next/today), and action buttons matching Figma node `7:1110`.
2. **View-mode toggle** uses a segmented pill control (`border-radius: 8px`, bg `#F1F5F9`) with active state (white bg, shadow) and inactive state (transparent) per Figma.
3. **Date navigation** shows prev/next icon buttons and a current date label (e.g. "January 2025") with a "Today" shortcut button.
4. **Add Task button** is primary blue (`#137FEC`), contains icon + "Add Task" text, and opens the task creation dialog.
5. **Left panel** renders at fixed 300px width with: panel header ("TASK NAME" uppercase), group section headers (expand/collapse chevron + group name), and task rows (status dot + task name).
6. **Group sections** are collapsible — clicking the chevron/header toggles child task rows.
7. **Task rows** in the left panel are 48px tall with `padding: 0 24px`, task name in Manrope Regular 400 / 14px / `#334155`, and a bottom border.
8. **Right panel (chart)** fills remaining width with a date header row (40px tall, date range labels) and a scrollable chart grid below.
9. **Date columns** display date ranges (e.g. "Oct 01 - 07" for week view) with vertical grid lines separating columns.
10. **Gantt bars** render as rounded rectangles (`border-radius: 6px`, `height: 28px`) with fill `#DBEAFE`, border `#BFDBFE`, and a shadow. Bar width is proportional to task duration.
11. **Gantt bar label** displays inside the bar in Manrope Medium 500 / 12px / `#1E40AF`.
12. **Synchronized scrolling** — vertical scroll of the left task list and right chart are synchronised. Horizontal scroll applies only to the chart area and its date header.
13. **Loading state** shows a skeleton placeholder matching the split-panel layout.
14. **Empty state** renders when no tasks have due dates, reusing `TimelineEmptyStateComponent` (updated for new layout if needed).
15. **vis-timeline usage updated** — If Approach A: vis-timeline renders only in the right chart panel with its built-in groups sidebar hidden; custom CSS theming matches Figma bar styling. If Approach B: `vis-timeline` / `vis-data` dependencies are fully removed.
16. **Data integration** uses `TaskService.getTimelineTasks()` to fetch tasks and groups them by `parentTaskId` (or flat list if no hierarchy).
17. **Existing routes and shared components are unaffected** — Shellbar, Sidebar, Dashboard, and other views continue to work.
18. **Typography** uses Manrope at specified weights/sizes throughout; column header uses Bold 700 / 12px / uppercase with `0.6px` letter-spacing.
19. **Sidebar active state** highlights "Timeline" nav item when on the `/timeline` route (verify existing `routerLinkActive` works correctly).

Use vis-timeline's `template` option for custom bar HTML if CSS-only overrides aren't sufficient. Handle edge cases: tasks extending beyond visible range, zero-duration tasks (render as diamonds/milestones), timezone normalization.
- The synchronized scroll can be achieved with a shared `scrollTop` binding between two `<div>` containers using `(scroll)` event handlers or `IntersectionObserver`. Consider using `scroll-behavior: smooth` for programmatic scrolls.
- The horizontal chart width should be dynamically calculated: `columnWidth × numberOfColumns`. For week view with 5 visible columns at 240px each = 1200px.
- Column widths per view mode:
  - **Day:** 60px per day (show individual dates, e.g. "Oct 01", "Oct 02")
  - **Week:** 240px per week (show ranges, e.g. "Oct 01 - 07")
  - **Month:** 200px per month (show month names, e.g. "October 2025")
- The "Today" indicator should be a subtle vertical line in the chart area highlighting the current date.
- The `TimelineEmptyStateComponent` can be reused with minimal styling updates for the new layout container.
- Bar colour coding by status (future enhancement): In Progress = blue (current Figma), To Do = gray, Done = green — reuse tokens from Story 4.10.
